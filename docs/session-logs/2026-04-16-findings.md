## FINDING-001 STATUS UPDATE: Confirmed bug at scale

**Status:** **Confirmed bug, not intentional.** Upgraded priority to HIGH.

**Pattern:** Fractional px sizes in DOM indicate developers used `em` / `rem`
with non-integer multipliers instead of explicit `px`. When the browser
computes these against a 16px root, non-integer pixel values leak into the
rendered layout.

**Evidence (from 2026-04-16 authenticated drift run):**
- `Nunito 11.223px / 500` — almost certainly `font-size: 0.7rem` (`0.7 × 16 = 11.2`;
  `11.223` is the sub-pixel result after some layout multiplication)
- `Nunito 13.333px / 400` — `0.833em × 16 = 13.328` (rounded to 13.333 by the
  CSSOM)
- `Nunito 14.4px / 600` — `0.9em × 16 = 14.4`

All three have a clear integer-px intent that exists in the canonical scale:
- `11.2px` → nearest canonical `12px` (typography.02/06/07/12)
- `13.3px` → nearest canonical `14px` (typography.03/08/09/13)
- `14.4px` → nearest canonical `14px` same

**Fix:** Replace relative font-size declarations in the student app with
explicit `px` values that exist in the canonical type scale. Either:
- Convert all `em` / `rem` font-sizes to `px` where the multiplier is non-integer
- Or use `rem` with strict integer multipliers of the 16px root (0.625rem, 0.75rem,
  0.875rem, 1rem, 1.125rem, 1.25rem) — which resolve to 10/12/14/16/18/20px exactly

**Priority:** **HIGH.** The bug is systematic — three distinct fractional
sizes showed up on three pages, implying this is a codebase-wide pattern, not
a one-off. Readability and type-scale consistency both suffer.

## FINDING-006: Circular spacing/radius comparison (backlog)

**Status:** Backlog — not blocking current drift review but makes today's
spacing/radius numbers partially self-referential.

**Cause:** The resolver's `passthroughTier` promotes any DOM-extracted token
to confidence 0.9 (canonical) if it comes in at confidence 0.9 from the
extractor. DOM tokens default to 0.9. So every DOM spacing/radius value
becomes "canonical" in the ledger.

**Effect:** When `drift:detect` runs, spacing/radius DOM tokens are compared
against ledger canonicals — but those ledger canonicals were themselves
DOM-sourced. DOM matches DOM. Not actual drift against design intent.

**Today's impact:** The 12 spacing matches and 9 radius matches in the
2026-04-16 drift report are partially self-matching. The 2 spacing drifts
(`25px` / `24px` against canonical `20px`) and 3 radius drifts (`16px` /
`24px` / `3px` against `20px` / `6px`) are DOM-vs-DOM comparisons — still
potentially useful signal (near-miss values the dev team might want to align),
but not drift in the design-intent sense.

**Fix options:**
1. Downgrade DOM-only spacing/radius tokens to `candidate` (confidence 0.5)
   by default in `passthroughTier`. Drift detection would then surface them
   for review rather than treating them as canonical.
2. Extend the Figma extractor to walk `layoutMode` / `itemSpacing` /
   `padding*` / `cornerRadius` on frames. Gives Figma its own spacing/radius
   canonicals to compare against. Better solution, bigger scope.

**Priority:** Medium. Not blocking the publish pipeline (spacing/radius
tokens aren't in `tokens.css` output yet), but should be fixed before the
drift review dashboard ships.

## FINDING-007: Identical element counts across student URLs (open)

**Status:** Open. Also observed yesterday (FINDING-004). Hydration wait didn't
fix it; not an auth problem either (auth now works).

**Pattern:** Extractor walks 700/707/707 elements across `/my-feed/`,
`/courses/`, `/profile/` — essentially identical. 532 visible elements on
each. Same 39 distinct colors.

**Likely cause:** The student app is a client-side-routed SPA with a
persistent shell (nav, header, sidebar). Route changes swap content panels
within a stable DOM tree. Element count stays roughly equal because only
~50–100 nodes change per route, not the full tree.

**To verify next session:** Add a per-page probe that logs
`window.location.href` after the post-navigation wait, plus the text content
of the main content area. Confirms whether the router actually navigated or
the extractor is stuck on one view.

**Priority:** Medium. If the content DOES change per route but the token
surface largely doesn't, the drift report is still valid (we're capturing the
app's overall token footprint). If the router is broken in the crawler, we're
missing real per-page variation.

## UPDATE: Unknown color `#4a3fb4` reclassified (2026-04-16 evening)

Manual DOM inspection of the feed tab toggle revealed `#4a3fb4` is the
**active state background** for the Global Feed / My Feed tabs
(class: `togglebutton_activePaid`). This is **not a library leak** — it's an
intentional UI state color that the Figma extractor never captured because
the Figma source had no named styles at all.

**Action taken:** Added to `ledger/manual-canonicals.json` as
`color/interactive/tab-active` at confidence 0.85.

**Impact on previous drift report (2026-04-16):** Unknown color count
**drops from 5 to 4.** Remaining unknown colors needing design triage:

- `#4bae4f` — still suspect (close to Material Green 500 `#4CAF50`)
- `#6c2bd9` — deep purple variant
- `#963e02` — unusual brown/orange
- `#ab9dff` — light lavender

Additionally: `radius/pill` (`30px`) was discovered during the same
inspection as the tab toggle's border-radius. Added to manual-canonicals
at confidence 0.85 — prevents future drift reports from flagging it as
"unknown radius."
