# Session Findings — 2026-04-15

First end-to-end drift report shipped for the student surface. These findings
are from the value-based comparison of DOM tokens (live `champ.brightchamps.com`)
against the canonical ledger synthesized from the student Figma file.

---

## FINDING-001: Fractional font sizes in DOM

**Cause:** Developers used relative units (`em` / `rem` / unitless) where Figma
specified integer `px` values. Computed styles resolve to non-integer pixel
values that never existed in the design.

**Evidence:**
- `Nunito 13.3333px` (≈ `0.833em × 16px base`)
- `Nunito 11.2px` (≈ `0.7em × 16px base`)
- Both seen on `/my-feed/`

**Fix:** Convert all text-size declarations in the student app to `px`, or to
`rem` with integer multipliers of the root font size. Audit the CSS / styled-
components / theme for any fractional multipliers.

**Priority:** **Medium.** Visual impact is small (half-pixel rendering varies
by browser and display DPR), but design-system hygiene is high — fractional px
can't map to any canonical typography token, so every use is flagged as drift.

---

## FINDING-002: Display type sizes missing from canonical

**Cause:** The canonical typography scale tops out at `20px` because that was
the highest-used size in the student Figma file during extraction. Live DOM
contains larger display text that was never captured as a named type style in
Figma.

**Evidence:**
- `Nunito 32px / 400` on `/my-feed/` (hero heading)
- `Nunito 32px / 900` on `/my-feed/` (numeric display)

**Fix:** Design team decision. Two options:
1. **Add to canonical** — extend type scale with `display-32-400` and
   `display-32-900`. Bump `TYPO_CANONICAL_TOP_N` from 16 toward 18.
2. **Flag as intentional off-system** — hero/display text is a legitimate
   one-off class, not a drift. Document in the design system site as "use
   sparingly" without canonicalizing.

**Priority:** **Low.** Blocks only hero/display components. Non-blocking for
body text, buttons, labels — those all hit canonical or drift cleanly.

---

## FINDING-003: System-gap — translucent overlays missing from Figma source

**Cause:** The student Figma file contains no named effect styles. Overlays,
scrims, and dividers that the live app uses (semi-transparent dark neutrals)
were never captured during extraction.

**Evidence:**
- `#33333340` — dark gray at 25% alpha (modal backdrops / section dividers)
- `#0000001a` — black at 10% alpha (soft shadows / card surfaces)

**Resolution (this session):** Added both as canonical tokens in the ledger
with `confidence: 0.8` per the DOM-only rule (admin pattern — no Figma backing,
DOM is source of truth). Names chosen:
- `color/overlay/dark` = `#33333340`
- `color/overlay/subtle` = `#0000001a`

**Followup needed:** These manual additions currently live only in
`ledger/tokens.json`. A future `npm run ledger:build` will overwrite them.
Need a `ledger/manual-canonicals.json` overrides file that the resolver reads
and merges during build. See HANDOFF.md → "Known gaps".

**Priority:** **Medium.** Must be fixed before any future rebuild or the two
overlay tokens will vanish.

## FINDING-004: SPA hydration not completing before DOM extraction

**Cause (hypothesis):** Playwright's `waitUntil: 'networkidle'` fires when
network traffic is quiet for 500ms, but the student SPA continues client-side
hydration after that window closes. The extractor walks a partially hydrated
DOM and misses content-bearing elements.

**Evidence:**
- Three distinct student URLs (`/my-feed/`, `/courses/`, `/profile/`) returned
  **identical** element counts: 147/150/150 walked, 64 visible on each
- The aggregate token count after three URLs (8 colors, 16 typography, 30
  spacing, 5 radius, 2 shadows) is unchanged from a single-URL run. Three
  pages of actual content would normally produce a noticeably larger pool.

**Fix options:**
1. Replace `networkidle` with a targeted wait — e.g. `page.waitForSelector(...)`
   keyed to a known-hydrated element per surface
2. Add a fixed `page.waitForTimeout(3000)` after `networkidle` (simple but
   fragile)
3. Use Playwright's `waitForLoadState('networkidle')` and then a re-check
   after 2s to confirm element count stabilized

**Priority:** **High.** Blocks meaningful multi-page drift detection. Every
URL added to the student surface currently returns the same shell. Until this
is fixed, extending URL lists has no effect on coverage.
